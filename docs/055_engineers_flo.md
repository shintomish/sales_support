# 技術者メール フロー説明書

> 対象読者：当社営業担当者

---

## 1. 全体の流れ

```
Gmail受信
  ↓（自動・15分毎）
メール取込（emailsテーブルに保存）
  ↓（自動・15分毎）
技術者メール or 案件メール に自動分類
  ↓（自動・1時間毎・50件バッチ）  ← 2026-04-09 自動化
技術者メール画面に表示・スコアリング
```

---

## 2. 「技術者メール」に分類されるタイミング

**自動で15分毎に動きます。**

Gmailに新着メールが届いてから最大30分以内（取込15分 + 分類15分）に技術者メール画面へ反映されます。

### 技術者メールと判定される条件（いずれか一つ該当で技術者メール）

| 優先順 | 条件 | 例 |
|--------|------|----|
| 1位 | スキルシート等の **添付ファイルあり** | PDF・Excel・Word が添付されている |
| 2位 | 件名に **【技術者情報】** が含まれる | 件名「【技術者情報】N.S様のご紹介」 |
| 3位 | 件名に **人材系キーワード** が含まれる | 人材・人財・要員・スキルシート・経歴書・職務経歴・フリーランス・ご紹介・弊社直・弊社要員・弊社社員・正社員・プロパー |

### 案件メールと判定される条件

上記に該当しない場合は案件メールになります。

---

## 3.「新着取込」の動作

### 自動実行（2026-04-09 より）

**1時間毎に自動で最大50件を処理します。**

Gmailに新着メールが届いてから最大約1時間45分以内（取込15分 + 分類15分 + 新着取込最大75分）に技術者メール画面へ反映されます。

画面右上の「新着取込」ボタンを押すと即時処理も可能です（50件上限）。

### 対象になるメール
- `category = engineer`（技術者メールと分類済み）
- かつ、まだ技術者メール画面に登録されていないもの（初回処理のみ）

### 対象から外れるメール（＝処理されない）
- すでに技術者メール画面に一度でも表示されたことがあるメール
  → 新着取込は「初回のみ」が対象です

---

## 4.「全件再スコア」ボタンの対象データ

**画面の「全件再スコア」ボタンを押したとき（手動のみ）。**

### 対象になるデータ
- 技術者メール画面に登録されている **すべてのレコード**（新規・要確認・登録済・提案中・稼働中・除外 すべて含む）

### 何が更新されるか
- 氏名・最寄り駅・稼働開始・スキルなどの **抽出情報を再計算**
- スコア（点数）と **ステータス（新着 / 要確認 / 除外）を再判定**

### 対象から外れるデータ
- なし（全件が対象）

### 添付ファイルの扱い
- 全件再スコア時は **添付ファイル（スキルシート）の解析をスキップ** します
  - 理由：数千件分のファイルをダウンロードするとタイムアウトするため
  - メール本文のみで再抽出されます

> 💡 **いつ使うか：** 抽出ロジックを改善した後に、過去データも正しい情報に更新したいときに使います。

## 6. ステータス変更ボタンの場所

| ステータス | 変更できる場所 |
|-----------|--------------|
| 技術者確定 / 非対象（除外） | リスト行を展開したカード内のボタン |
| 登録済 / 除外 | 右ペイン（レコード選択時）のステータスボタン |
| 提案中 | 右ペイン（登録済の状態で選択時）のステータスボタン |
| 稼働中 | 右ペイン（提案中の状態で選択時）のステータスボタン |

> 提案中・稼働中への変更は右ペインでレコードを選択した状態でないと操作できません。

### 「技術者として登録→」ボタンについて（2026-04-09 追加）

右ペインの「技術者として登録→」ボタンを押すと、抽出情報（氏名・最寄駅・スキル等）が引き継がれた状態で技術者登録画面が開きます。

登録フォームで「登録する」を押して完了すると、**技術者メールのステータスが自動で「登録済」に変わり**、技術者メール画面に戻ります。

```
「技術者として登録→」クリック
  ↓
技術者登録画面（抽出情報が自動入力）
  ↓「登録する」クリック
技術者DBに保存
  ↓（自動）
技術者メールのステータス → 登録済
  ↓
技術者メール画面に戻る
```

## 7. ステータスの意味

| ステータス | 意味 | スコア目安 |
|-----------|------|-----------|
| 新着 | 技術者メールと確定（スコア高） | 60点以上 |
| 要確認 | 技術者メールかもしれない（要目視確認） | 40〜59点 |
| 除外 | 技術者メールではないと判定 | 39点以下 |
| 技術者確定 | 営業が「技術者確定」ボタンを押した | — |
| 提案中 | 案件に提案中 | — |
| 稼働中 | 現在稼働中 | — |

## 8. スコア計算の詳細

スコアは **0〜100点** で自動計算され、点数によってステータスが決まります。

### ステータス判定基準

| スコア | ステータス | 意味 |
|--------|-----------|------|
| 60点以上 | **新着** | 技術者メールとほぼ確定 |
| 40〜59点 | **要確認** | 技術者メールの可能性あり・目視確認が必要 |
| 39点以下 | **除外** | 技術者メールではないと判定 |

---

### スコア計算式

#### ステップ1：除外チェック（スコア計算前に強制除外）

以下に該当する場合はスコア0・除外確定となり、スコア計算は行いません。

| 除外条件 | 内容 |
|---------|------|
| 件名に除外ワード | 配信停止・メルマガ・広告・請求書・お支払い・ご挨拶・お知らせ |
| 送信元アドレス | no-reply・noreply を含む |
| 自社ドメイン | aizen-sol.co.jp からのメール |

#### ステップ2：加点計算（最大85点設計）

| 区分 | 加点 | 条件 |
|------|------|------|
| **[A] 明示的紹介ワード** | +15点 | 件名または本文に以下のいずれか1つでも含まれる：スキルシート・経歴書・職務経歴書・要員ご紹介・技術者ご紹介・人材ご紹介・エンジニアご紹介・技術者情報・要員情報・エンジニア情報 |
| **[B] 稼働条件ワード** | +10点 | 件名または本文に以下のいずれか1つでも含まれる：稼働開始・稼働可能・空きあり・対応可能・稼働率・即稼働・即日対応・参画可能 |
| **[C] 技術スタック** | +3点/件（上限20点） | 下記の技術キーワードが1件につき3点、最大20点（約7件分） |
| **[D] 所属区分ワード** | +10点 | 本文に以下のいずれか1つでも含まれる：BP・フリーランス・自社社員・下請け・一社先・個人事業主・契約社員 |
| **[E] 添付ファイルあり** | +15点 | スキルシート等の添付ファイルがある |

**合計最大：15 + 10 + 20 + 10 + 15 = 70点**（実装上の上限は100点にクランプ）

| カテゴリ | キーワード |
|---------|-----------|
| 言語 | Java / PHP / Python / C# / Ruby / Go / Golang / Swift / Kotlin |
| FW | Spring / SpringBoot / Laravel / Django / Flask / .NET / Rails / React / Vue / Angular / TypeScript / JavaScript |
| クラウド | AWS / EC2 / RDS / S3 / Lambda / Azure / GCP |

| DB | MySQL / PostgreSQL / Oracle / SQLServer / MongoDB / Redis |

|------|---------|------|

## 9. 設計経緯メモ（開発者向け）

### なぜ当初、新着取込を自動化しなかったか

技術者メールの新着取込は、1件ごとに以下の重い処理が走る：

1. **Gmail API からスキルシート（PDF/Excel）をダウンロード**
2. **PhpSpreadsheet / PdfParser でテキスト抽出**
3. **Claude API に送信して氏名・スキル・最寄駅などを解析**（APIコスト発生）

これを15分毎に自動実行すると、メールが大量になるとタイムアウトが発生した（実際に経験済み）。またGmail APIのレート制限やClaude APIコストの超過リスクもあった。そのため当初は手動ボタンのみとし、50件バッチ処理を導入してタイムアウトを回避していた。

案件メールのスコアリングは正規表現による本文マッチのみ（外部API不要・処理が軽い）のため15分毎の自動実行が可能だった。

### なぜ2026-04-09に自動化したか

- 50件バッチ処理が安定稼働していることが確認できた
- 毎回手動でボタンを押す運用が営業負担になっていた
- 1時間毎・50件バッチであれば外部API負荷・コストも許容範囲と判断
- 手動ボタンも残しているため即時処理も引き続き可能
