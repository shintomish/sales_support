# SESマッチング機能設計書

## 1. 概要

### 1.1 目的
1日5,000通規模のアウトソース（案件）メールを資産として活用し、営業の提案速度と成約率を向上させる。

### 1.2 基本思想
- `emails`テーブル（既存・保管庫）を起点に、**案件業務エンジン**を新設する
- AI抽出は段階導入（初期はルールベース、後にAI差し替え可能な設計）
- 最終判断は営業。スコアは「根拠を出す道具」であり「結論」ではない

### 1.3 全体アーキテクチャ

```
Gmail
  ↓ sync（15分毎）
emails テーブル（保管庫）
  ↓ 案件判定スコアリング
project_mail_sources テーブル（案件業務エンジン）
  ↓ マッチング
engineers テーブル（技術者）
  ↓
提案アクション（返信メール生成）
```

## 2. 案件メール判定・取込機能

### 2.1 処理フロー

```
① 除外判定（即NG）
② 強ワード判定（即OK）
③ スコア判定（グレーゾーン）
④ 案件 or 非案件の確定
```

### 2.2 ① 除外判定（最優先・処理負荷最小）

以下のいずれかに該当する場合、即座に非案件とする（本文解析なし）。

| 種別 | 条件 |
|------|------|
| メルマガ | 件名に「配信停止」「メルマガ」「広告」 |
| 請求系 | 「請求書」「お支払い」 |
| 採用系 | 「正社員募集」「中途採用」 |
| 事務連絡 | 「ご挨拶」「ご連絡」「お知らせ」 |
| 自動送信 | 差出人が `no-reply` / `noreply` |

### 2.3 ② 強ワード判定（即OK）

以下の条件を満たす場合、即座に案件確定。

- **Aグループ**（業界用語）のいずれか1語 AND
- **BまたはC**のいずれか1語

| グループ | キーワード |
|---------|-----------|
| A（業界用語） | 案件、要員、技術者募集、エンジニア募集、SE募集、PG募集 |
| B（条件系） | 単価、スキル、勤務地、最寄駅、稼働、開始 |
| C（契約系） | 準委任、常駐、リモート |

### 2.4 ③ スコア判定（グレーゾーン：40〜59点が要確認）

#### スコアリング辞書

**技術キーワード（上限30点）**

| カテゴリ | キーワード例 | 加点 |
|---------|------------|------|
| 言語/FW | Java, SpringBoot, PHP, Laravel, Python, Django, C#, .NET, JavaScript, TypeScript, React, Vue | +20 |
| インフラ/クラウド | AWS, Azure, GCP, Docker, Kubernetes, Linux | +15 |
| DB | MySQL, PostgreSQL, Oracle, SQLServer, MongoDB | +10 |

**工程キーワード**

| カテゴリ | キーワード例 | 加点 |
|---------|------------|------|
| 上流 | 要件定義、基本設計、詳細設計 | +15 |
| 開発 | 開発、実装、製造 | +10 |
| その他 | テスト、保守、運用 | +5 |

**金額・期間キーワード**

| カテゴリ | 条件 | 加点 |
|---------|------|------|
| 金額（具体） | `\d{2,3}万円` / `\d{2,3}万` | +20 |
| 期間・稼働 | 月、人月、即日、長期 | +10 |
| 金額（曖昧）| スキル見合い、応相談 | -10 |

**場所・書式**

| 条件 | 加点 |
|------|------|
| 都市名（東京・品川・在宅等） | +10 |
| 日付表現（即日・5月〜） | +10 |
| 表形式（箇条書き・罫線） | +10 |

#### 判定基準

| スコア | 判定 |
|-------|------|
| 60点以上 | 案件確定 |
| 40〜59点 | 要確認（営業が判断） |
| 39点以下 | 非案件 |

#### 判定結果の形式（AI差し替えを見据えた共通IF）

```json
{
  "score": 58,
  "reasons": ["Java keyword matched", "No price found"],
  "engine": "rule"
}
```

### 2.5 ④ ドメイン学習（蓄積後に有効化）

BP会社（ドメイン）ごとの案件率を補正値として利用。

| 案件率 | スコア補正 |
|-------|-----------|
| 80%以上 | +20点 |
| 20%以下 | -20点 |

| 項目 | 点数 | 備考 |

| ② スキル適合度 | 25点 | |

| ④ 稼働・タイミング | 10点 | |

### 4.2 ① 必須条件一致（40点・足切り）

国籍・年齢・勤務地/リモート可否。致命的NGは即除外（表示しない）。

| 評価 | 点数 |
|-----|------|
| 完全一致 | 40 |
| 一部NG | 20 |
| 致命的NG | 0（非表示） |

### 4.3 ② スキル適合度（25点）

**必須スキル（15点）**

| 一致率 | 点数 |
|-------|------|
| 100% | 15 |
| 80% | 12 |
| 50% | 8 |
| 0% | 0 |

**尚可スキル（10点）**

| 一致数 | 点数 |
|-------|------|
| 2以上 | 10 |
| 1 | 5 |
| 0 | 0 |

### 4.4 ③ 条件適合度（20点）

**単価（10点）**

| 差分 | 点数 |
|-----|------|
| 希望内 | 10 |
| +5万以内 | 7 |
| +10万以内 | 4 |
| 超過 | 0 |

**勤務地（5点）** / **契約形態（5点）**

### 4.5 ④ 稼働・タイミング（10点）

| 状態 | 点数 |
|-----|------|
| 即日可 | 10 |
| 2週間以内 | 7 |
| 1ヶ月以内 | 4 |
| 未定 | 0 |

### 4.6 ⑤ 商流・実績補正（5点）

同顧客実績あり +3点、同業界実績あり +2点

### 4.7 スコアランク

| スコア | 表示 | 意味 |
|-------|------|------|
| 85〜100 | ◎ 即提案 | そのまま出す |
| 70〜84 | ○ 有力 | 条件微調整 |
| 55〜69 | △ 検討 | 交渉前提 |
| 〜54 | × 見送り | 原則出さない |

## 5. UI設計

### 5.1 要確認メール一覧（営業が毎朝確認）

スコア40〜59点のメールを一覧表示。1行1判断。

```
［要確認案件］（15件）

□ 55点｜Java／AWS｜70万？｜品川｜BP-A
□ 48点｜PHP？｜単価記載なし｜横浜｜BP-B
```

詳細画面では判定理由・本文ハイライト・ワンクリック判断（案件にする / 非案件 / 後で）。

### 5.2 案件→技術者マッチング画面（3秒判断）

**上部（案件サマリー・固定）**

```
Java／Spring／AWS
75〜80万｜品川（常駐）｜5月〜
外国籍不可｜年齢〜45
```

**中央（マッチング結果）**

```
◎ 即提案（最大3名）
  [92点] 田中｜Java5y AWS｜即日｜78万  ✔ 条件一致
  [88点] 佐藤｜Java4y    ｜2週 ｜75万  ▲ 単価▲3万

○ 有力（最大5名）
  [76点] 鈴木｜Java3y    ｜即日｜82万  ▲ 単価＋7万

△ 検討（折りたたみ・デフォルト非表示）
```

**操作（1技術者につき2択）**

`[提案]` `[除外]`

**技術者クリック時（スコア内訳モーダル）**

```
必須条件    40/40
スキル      22/25（AWSなし）
条件        14/20（単価+5万）
稼働        10/10
──────────
合計        86点
```

### 5.3 色・視線誘導

| 要素 | 色 |
|-----|---|
| ◎ 即提案 | 緑 |
| ○ 有力 | 青 |
| △ 検討 | グレー |
| NG条件 | 赤 |
| 減点理由 | オレンジ |

## 7. AI導入フェーズ設計

| フェーズ | 内容 |
|---------|------|
| フェーズ1（現在） | ルール100%。営業修正ログを蓄積 |
| フェーズ2 | AIで「要確認（40〜59点）」のみ再判定 |
| フェーズ3 | 低スコア自動除外・高スコア自動案件化 |

判定IFを共通化することで、`engine: "rule"` → `engine: "ai"` の差し替えのみで移行可能。
