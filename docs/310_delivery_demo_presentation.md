# 配信管理・送信履歴 機能説明資料

**対象**: 社内営業向け デモ説明  
**作成日**: 2026-04-12  
**最終更新**: 2026-04-29  
**説明者**: 新冨 泰明

---

## 1. 背景と目的

SES営業では、案件メールへの紹介メール送信後に「誰に送ったか」「返信がきたか」を把握するのが難しい。

本機能は以下の2点を解決する。

| 課題 | 解決策 |
|------|--------|
| 一斉送信した後、個別の反応が追えない | **配信管理** で送信キャンペーンごとに反応を一覧管理 |
| 個別メール・一斉メール合わせた全送信履歴が散らばる | **送信履歴** で全送信を一画面で検索・確認 |

---

## 2. 配信管理（`/deliveries`）

### 2-1. 概要

一斉メール配信（キャンペーン）の作成・管理を行う画面。4つのタブで構成。

| タブ | 役割 |
|------|------|
| 配信先一覧 | 送信先のメールアドレス一覧・管理（一括有効/無効・状態スナップショット保存/復元対応） |
| 一斉配信履歴 | 過去の一斉送信の実績を確認 |
| 提案スレッド | 案件メール／技術者メール単位で送受信履歴をスレッド表示（返信有無・未読バッジ） |
| 新規配信 | メール本文を入力して一斉送信を実行 |

---

### 2-2. 一斉配信履歴タブ

#### 検索・絞り込み

- **キーワード検索**: 件名・紐づき案件名・受信者名・受信者メールアドレスで絞り込み
- **日付範囲**: 送信日 From〜To でフィルタ
- **送信種別**（`send_type`）／送信者でも絞り込み可
- ソート: 送信日時・件名・送信者・案件名（全列ソート対応）
- 検索結果件数をリアルタイム表示

#### 一覧表示項目

| 列 | 内容 |
|----|------|
| 件名 | 送信メールの件名。紐づき案件があればその名称も表示 |
| 送信者 | 送信を実行したユーザー名 |
| 送信日時 | 送信実行日時 |
| 送信数 | 送信アドレスの総数 |
| 成功 / 失敗 | 送信結果の内訳 |
| 返信数 | `replied_count`（受信メールから自動紐付け） |

---

### 2-3. キャンペーン詳細（`/deliveries/campaigns/[id]`）

「詳細」ボタンで個別キャンペーンの内訳を確認。

#### サマリーカード

- 件名・送信者・送信日時・紐づき案件
- 送信数 / 成功 / 失敗 / **返信あり** 件数
- **返信率** と進捗バー

#### 送信履歴テーブル

| 列 | 内容 |
|----|------|
| 名前 | 受信者の担当者名 |
| メールアドレス | 受信者アドレス |
| 状態 | 送信済 / 返信あり / 失敗 |
| 返信 | 返信がある場合はクリックで返信本文をプレビュー（`reply_email_id` で自動紐付け） |

- メール / 名前でのインクリメンタルサーチ
- 状態（送信済 / 返信あり / 失敗）でのフィルタ

---

## 3. 送信履歴の統合（旧 `/send-histories`）

> **2026-04-13 廃止**: 独立した「送信履歴」画面（`/send-histories`）は廃止し、
> すべての送信記録を **配信管理**（`delivery_campaigns` + `delivery_send_histories`）に一元化。
> 一斉配信／個別提案／マッチング提案／技術者メール提案 を `send_type` で識別する。

### 3-1. 送信種別（`send_type`）

| `send_type` | 表示名 | 発生元画面 | 紐付け情報 |
|-------------|--------|------------|------------|
| `delivery` | 一斉配信 | `/deliveries`（新規配信タブ） | — |
| `proposal` | 個別提案 | `/project-mails` | engineer_id |
| `matching_proposal` | マッチング提案 | `/matching/[id]` | engineer_id / public_project_id |
| `engineer_proposal` | 技術者メール提案 | `/engineer-mails` | engineer_mail_source_id / public_project_id |

### 3-2. 一覧での確認方法

- 「一斉配信履歴」タブ: `delivery` のみを一覧表示（`exclude_proposals` フィルタ）
- 「提案スレッド」タブ: `proposal` / `engineer_proposal` / `delivery`（案件・技術者メール紐付きあり）をスレッド単位で集約
- いずれもキーワード（件名・案件名・宛先メール／名前）・送信日 From〜To・送信者・送信種別で絞り込み可能

## 4. 返信紐づけ機能（実装済み）

### 4-1. 概念図

```
送信メール（delivery_send_histories）
    ↓ email
    ↕ 照合（from_address が一致）
受信メール（emails）
    → reply_email_id / replied_at を自動セット
```

### 4-2. 紐づけロジック

1. Gmail から受信した返信メールの `from_address` を取得
2. 送信済メールの宛先（`delivery_send_histories.email`）と照合
3. 一致した場合、送信レコードに `replied_at` と `reply_email_id` を記録し、ステータスを「返信あり」に更新

### 4-3. 画面での確認

- **キャンペーン詳細**: 「返信あり」行をクリック → 返信本文プレビュー
- **提案スレッドタブ**: スレッド単位で送受信を時系列表示。未読返信があるスレッドにはバッジを表示

## 7. 営業からのフィードバック確認ポイント

- [ ] 返信紐づけの表示単位は「キャンペーン単位」か「個人単位」か、どちらが使いやすいか
- [ ] 返信率の閾値アラート（例: 返信率30%未満の場合に通知）は必要か
- [ ] 送信履歴に「案件名」「技術者名」列を追加すべきか（現在はマッチング提案のみ対象）
- [ ] 一斉送信の宛先グループ（業種別・地域別など）の切り分けは必要か
