# 技術者メール フロー説明書

> 対象読者：当社営業担当者
> 最終更新：2026-04-29（新着取込スケジュールを15分毎100件に同期、新着取込ボタン廃止を反映）

---

## 1. 全体の流れ

```
Gmail受信 / KAGOYA IMAP受信
  ↓（自動・各15分毎）
メール取込（emailsテーブルに保存）
  ↓（自動・15分毎）
技術者メール or 案件メール に自動分類
  ↓（自動・15分毎・100件バッチ）
技術者メール画面に表示・スコアリング
```

---

## 2. 「技術者メール」に分類されるタイミング

**自動で15分毎に動きます。**

Gmailに新着メールが届いてから最大30分以内（取込15分 + 分類15分）に技術者メール画面へ反映されます。

### 技術者メールと判定される条件（いずれか一つ該当で技術者メール）

| 優先順 | 条件 | 例 |
|--------|------|----|
| 1位 | スキルシート等の **添付ファイルあり** | PDF・Excel・Word が添付されている |
| 2位 | 件名に **【技術者情報】** が含まれる | 件名「【技術者情報】N.S様のご紹介」 |
| 3位 | 件名に **人材系キーワード** が含まれる | 人材・人財・要員・スキルシート・経歴書・職務経歴・フリーランス・ご紹介・弊社直・弊社要員・弊社社員・正社員・プロパー |

### 案件メールと判定される条件

上記に該当しない場合は案件メールになります。

---

## 3.「新着取込」の動作

### 自動実行

**15分毎に自動で最大100件を処理します。**（2026-04 リファクタで 1時間毎50件 → 15分毎100件に統一）

Gmailに新着メールが届いてから最大45分以内（取込15分 + 分類15分 + 新着取込15分）に技術者メール画面へ反映されます。

> 「新着取込」ボタンは2026-04 に廃止しました。スケジューラに統一されています。

### 対象になるメール
- `category = engineer`（技術者メールと分類済み）
- かつ、まだ技術者メール画面に登録されていないもの（初回処理のみ）

### 対象から外れるメール（＝処理されない）
- すでに技術者メール画面に一度でも表示されたことがあるメール
  → 新着取込は「初回のみ」が対象です
- 過去にスコアリング処理が異常終了したメール（`storage/app/score_skip/{email_id}.skip` が残存）

## 4.「全件再スコア」ボタンの対象データ

**画面の「全件再スコア」ボタンを押したとき（手動のみ）。**

### 対象になるデータ
- 技術者メール画面に登録されている **すべてのレコード**（新規・要確認・登録済・提案中・稼働中・除外 すべて含む）

### 何が更新されるか
- 氏名・最寄り駅・稼働開始・スキルなどの **抽出情報を再計算**
- 希望単価・年齢も再抽出（2026-04-13 追加）
- スコア（点数）と **ステータス（新着 / 要確認 / 除外）を再判定**

### 対象から外れるデータ
- なし（全件が対象）

### 添付ファイルの扱い
- 全件再スコア時は **添付ファイル（スキルシート）の解析をスキップ** します
  - 理由：数千件分のファイルをダウンロードするとタイムアウトするため
  - メール本文のみで再抽出されます

### 処理件数・タイムアウト（2026-04-13 更新）

| | 変更前 | 変更後 |
|---|---|---|
| 処理件数上限 | 500件 | **300件/バッチ（offset繰り返しで全件）** |
| タイムアウト上限 | 300秒 | 120秒/バッチ |
| 添付ファイル解析 | なし | なし（本文のみ） |

> APIは1リクエストにつき300件処理し、`remaining` が0になるまでフロントから繰り返し呼ぶ実装です。実装上は `set_time_limit(120)` / `memory_limit=512M`。
> 添付ファイル解析をスキップするため、3,000件以上でも数十秒で完了します。

> 💡 **いつ使うか：** 抽出ロジックを改善した後に、過去データも正しい情報に更新したいときに使います。

> ⚠️ 低単価・単価未記載の自動除外ロジック（第11章）は、全件再スコア時には再適用されません（スコア点数のみでステータスを再判定）。

## 6. ステータス変更ボタンの場所

| ステータス | 変更できる場所 |
|-----------|--------------|
| 技術者確定 / 非対象（除外） | リスト行を展開したカード内のボタン |
| 登録済 / 除外 | 右ペイン（レコード選択時）のステータスボタン |
| 提案中 | 右ペイン（登録済の状態で選択時）のステータスボタン |
| 稼働中 | 右ペイン（提案中の状態で選択時）のステータスボタン |

> 提案中・稼働中への変更は右ペインでレコードを選択した状態でないと操作できません。
> タブを切り替えると右ペインの選択は自動でリセットされます（2026-04-13 改善）。

### 「技術者として登録→」ボタンについて（2026-04-09 追加）

右ペインの「技術者として登録→」ボタンを押すと、抽出情報（氏名・最寄駅・スキル等）が引き継がれた状態で技術者登録画面が開きます。

登録フォームで「登録する」を押して完了すると、**技術者メールのステータスが自動で「登録済」に変わり**、技術者メール画面に戻ります。

```
「技術者として登録→」クリック
  ↓
技術者登録画面（抽出情報が自動入力）
  ↓「登録する」クリック
技術者DBに保存
  ↓（自動）
技術者メールのステータス → 登録済
  ↓
技術者メール画面に戻る
```

## 8. スコア計算の詳細

スコアは **0〜100点** で自動計算され、点数によってステータスが決まります。

### ステータス判定基準

| スコア | ステータス | 意味 |
|--------|-----------|------|
| 60点以上 | **新着** | 技術者メールとほぼ確定 |
| 40〜59点 | **要確認** | 技術者メールの可能性あり・目視確認が必要 |
| 39点以下 | **除外** | 技術者メールではないと判定 |

---

### スコア計算式

#### ステップ1：除外チェック（スコア計算前に強制除外）

以下に該当する場合はスコア0・除外確定となり、スコア計算は行いません。

| 除外条件 | 内容 |
|---------|------|
| 件名に除外ワード | 配信停止・メルマガ・広告・請求書・お支払い・ご挨拶・お知らせ |
| 送信元アドレス | no-reply・noreply を含む |
| 自社ドメイン | aizen-sol.co.jp からのメール |

#### ステップ2：加点計算（最大85点設計）

| 区分 | 加点 | 条件 |
|------|------|------|
| **[A] 明示的紹介ワード** | +15点 | 件名または本文に以下のいずれか1つでも含まれる：スキルシート・経歴書・職務経歴書・要員ご紹介・技術者ご紹介・人材ご紹介・エンジニアご紹介・技術者情報・要員情報・エンジニア情報 |
| **[B] 稼働条件ワード** | +10点 | 件名または本文に以下のいずれか1つでも含まれる：稼働開始・稼働可能・空きあり・対応可能・稼働率・即稼働・即日対応・参画可能 |
| **[C] 技術スタック** | +3点/件（上限20点） | 下記の技術キーワードが1件につき3点、最大20点（約7件分） |
| **[D] 所属区分ワード** | +10点 | 本文に以下のいずれか1つでも含まれる：BP・フリーランス・自社社員・下請け・一社先・個人事業主・契約社員 |

## 9. 希望単価・年齢の自動抽出（2026-04-13 追加）

### 抽出タイミング

- **新着取込**（初回スコアリング）で自動実行
- **全件再スコア**ボタンで既存レコードも再抽出

### 対応フォーマット

| 本文例 | 抽出結果 |
|--------|---------|
| `■単金■\n60~70万円` | min=60, max=70 |
| `■単　金：110万（応相談）` | min=110, max=110 |
| `■単金：60万（応相談）` | min=60, max=60 |
| `■単価：〜80万` | min=80, max=80 |
| `■希望単価：70〜90万` | min=70, max=90 |
| `MN(46歳/女性)` | age=46 |
| `S.N(28歳/男性)` | age=28 |
| `年齢：35` | age=35 |

**対応済みの特殊パターン：**
- `■単　金` — 「単」と「金」の間の全角スペース
- `■単金■\n60~70万` — ■終端＋改行後に値、半角チルダ `~` の範囲指定
- `MN(46歳/女性)` — 括弧内の性別より前にある年齢

### フォームからの手動入力

自動抽出できなかった場合は、右ペインの編集フォーム（希望単価 下限・上限）に手動で入力して保存できます。

## 10. マッチ案件の単価フィルター（2026-04-13 追加）

技術者の希望単価が設定されている場合、**案件単価上限 >= 技術者希望単価** の案件のみ表示します。

```
表示条件：案件.unit_price_max >= 技術者の希望単価（unit_price_max または unit_price_min）
```

### 表示例（技術者希望単価 110万の場合）

| 案件 | 案件単価上限 | 表示 |
|------|------------|------|
| NWセキュリティプリセールスエンジニア | 123万 | 表示 ✓ |
| セキュリティ製品導入・設計支援 | 93万 | 非表示 |
| Javaバックエンドエンジニア | 75万 | 非表示 |

> 希望単価が未設定の場合は全案件を表示します。案件側に単価情報がない場合もその案件は表示されます。

## 11. 低単価・単価未記載の自動除外（2026-04-13 追加）

新着取込のスコアリング時に以下の条件で自動的に「除外」ステータスになります。

| 条件 | 除外理由タグ |
|------|------------|
| 希望単価の記載がない | `no_unit_price` |
| 希望単価が 35万円/月 未満 | `unit_price_too_low` |

業務方針：「案件側の提示金額より安い単金の技術者を紹介する」ため、単価が極端に低い・不明な場合は紹介対象外としています。

> 全件再スコア時はこの除外ロジックは再適用されません（スコア点数のみでステータスを再判定）。

## 12. 提案メール送信フォーム（2026-04-13 改善）

### 件名フォーマット

```
【技術者ご紹介】{案件タイトル}（max{案件単価上限}万円）
```

案件タイトル末尾に既に単価表示がある場合は除去して重複を防ぎます。

### 本文フォーマット

- 冒頭1行目：`{担当者名} 様`
- 2行目以降：AI生成の提案文（`{名前}（{年齢}歳）氏のご紹介です。` で始まる）
- 末尾：メール署名設定のテンプレートを自動挿入

### 送信フォームの項目

| 項目 | 内容 |
|------|------|
| 担当者名（宛名） | 入力すると本文冒頭の `○○ 様` をリアルタイムで書き換え |
| 宛先メール | 案件担当者のメールアドレス（自動入力・編集可） |
| 件名 | 上記フォーマットで自動生成・編集可 |
| 本文 | AIが生成・編集可 |
| 添付ファイル | ドラッグ＆ドロップまたはクリックで選択（複数可・最大10MB/件） |

| send_type | 送信元 | 紐付け情報 |
|-----------|--------|-----------|
| `delivery` | 案件メール → 一括配信 | — |

| `engineer_proposal` | 技術者メール画面からの提案 | engineer_mail_source_id / public_project_id |

## 15. 設計経緯メモ（開発者向け）

### なぜ当初、新着取込を自動化しなかったか

技術者メールの新着取込は、1件ごとに以下の重い処理が走る：

1. **Gmail API からスキルシート（PDF/Excel）をダウンロード**
2. **PhpSpreadsheet / PdfParser でテキスト抽出**
3. **Claude API に送信して氏名・スキル・最寄駅などを解析**（APIコスト発生）

これを15分毎に自動実行すると、メールが大量になるとタイムアウトが発生した（実際に経験済み）。またGmail APIのレート制限やClaude APIコストの超過リスクもあった。そのため当初は手動ボタンのみとし、50件バッチ処理を導入してタイムアウトを回避していた。

案件メールのスコアリングは正規表現による本文マッチのみ（外部API不要・処理が軽い）のため15分毎の自動実行が可能だった。

### なぜ2026-04-09に自動化したか

- 50件バッチ処理が安定稼働していることが確認できた
- 毎回手動でボタンを押す運用が営業負担になっていた
- 1時間毎・50件バッチであれば外部API負荷・コストも許容範囲と判断
- 手動ボタンも残しているため即時処理も引き続き可能

### 2026-04-13 の機能拡張

2026-04-13 に以下を追加した（本書 9〜14章）:

- 希望単価・年齢の自動抽出
- マッチ案件の単価フィルター（案件単価上限 >= 技術者希望単価のみ表示）
- 低単価・単価未記載の自動除外（業務方針に基づく紹介対象絞り込み）
- 提案メール送信フォームの改善（担当者名・添付DnD対応）
- 全件再スコアの上限撤廃（500件 → 全件、添付解析スキップで高速化）
- 送信履歴を配信管理に統合し `/send-histories` を廃止

これらは `550_engineer_mail_features.md` として別ファイルで管理していたが、2026-04-22 に本書へ統合し、550 は削除した。
