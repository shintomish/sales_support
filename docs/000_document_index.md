## 1. 番号体系ルール

| 番号範囲 | カテゴリ | 内容 |
|---|---|---|
| **000〜099** | プロジェクト基礎 | 背景・概要・体制・引き継ぎ・PM計画 |
| **100〜199** | 技術資料 | フロントエンド・バックエンド・インフラ構成 |
| **200〜299** | 運用手順書 | セットアップ・デプロイ・監視・バックアップ |
| **300〜399** | プレゼン・デモ資料 | 社内/顧客向け説明資料 |
| **400〜499** | 機能仕様・設計書 | feature_design / requirements / アーカイブ構想 |
| **500〜599** | 運用フロー説明書 | 営業担当向けマニュアル・Q&A |
| **600〜699** | 運用記録・ログ | 不達リスト・週次レポート・障害記録 |
| **900〜999** | 開発Tips・メモ | tmux・小ネタ・非公式メモ |

### 番号の使い方

- 1カテゴリ内で **10刻み** を基本 (010, 020, 030...)
- 関連する派生ドキュメントは **5刻み** で追加 (例: 420 本体 / 425 構想書)
- 新規追加時は既存番号の間に挿入せず、カテゴリ末尾に追加

## 4. 今後の運用ルール

### 命名規則
- ファイル名は `{3桁番号}_{英小文字スネークケース}.md`
- 日本語タイトルのドキュメントのみ `{3桁番号}_{日本語タイトル}.{拡張子}` を許可

例:
- OK: `050_pm_plan.md` / `010_プロジェクト背景.docx`
- NG: `050_PM計画書.md` (日本語+英数字混在)

### 更新時のフロー
1. ローカルで更新 (`~/sales_support/docs/`)
2. Google Drive にアップロード (または Google Doc 側で直接更新)
3. 両者の最終更新日を合わせる

### アーカイブ運用
- 設計変更で古くなった文書は削除せず、**「構想書」「アーカイブ」のサフィックス**を付けて残す
- 番号は新設計書に近い位置(+5刻み)に配置

### 新規追加時のチェック
1. 既存カテゴリに該当するか確認
2. 該当カテゴリの末尾番号 + 10 を使用
3. 本ドキュメント (`000_document_index.md`) に追記
