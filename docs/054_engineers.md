# 技術者メール機能 検討メモ（Q&A整理）

**打合せ日:** 2026-04-06

---

## 1. 案件メールに技術者メールのデータは入っているか

**入っていない。**

案件メール画面（`/project-mails`）は `category = 'project'` に分類されたメールのみを対象にしている。
`category = 'engineer'` のメールはスコアリングされず、案件メール画面には表示されない。

### メールの振り分けフロー（現状）

```
受信メール
  ↓ 自動分類（EmailClassificationService）
  ├─ category = 'project'  → スコアリング → 案件メール画面に表示
  └─ category = 'engineer' → スコアリングなし → メール画面（/emails）で確認するのみ
                             → 技術者管理への登録導線は現状なし
```

---

## 2. 技術者メール機能の新規実装について

### 実現可能性

**実装可能。** 案件メールの仕組みをそのまま雛形として使える。

### 案件メールとの対比

| | 案件メール（既存） | 技術者メール（新規） |
|---|---|---|
| 対象メール | `category = 'project'` | `category = 'engineer'` |
| 処理テーブル | `project_mail_sources` | `engineer_mail_sources`（新設） |
| スコアリング | `ProjectMailScoringService` | `EngineerMailScoringService`（新設） |
| 画面 | `/project-mails` | `/engineer-mails`（新設） |
| 最終アクション | 提案フロー → 成約 | 技術者管理への登録 |

### 実装で必要なもの

**バックエンド**
- `engineer_mail_sources` テーブル（migration）
- `EngineerMailScoringService`（スキルシート解析・スコアリングロジック）
- `EngineerMailController`（一覧・詳細・ステータス更新）

**フロントエンド**
- `/engineer-mails` ページ（案件メール画面に近いUI）
- 「技術者として登録」ボタン → `/engineers/create` にプリフィルして遷移

## 3. 打ち合わせヒアリング項目

実装に直結するため、以下を確認する。

### ① 振り分けロジック
- 添付ファイルあり（スキルシート）を優先するか
- 件名・本文のどんなキーワードで判定するか

### ② ステータスフロー
- 案件メールは `要確認 → 新着 → 提案済 → 面談 → 成約` だが、技術者メールは？
- 例：`要確認 → 登録済 → 提案中 → 稼働中` など

### ③ 抽出する情報
- スキルシート（PDF/Excel添付）からの情報抽出が必要か
- 本文テキストからどこまで自動抽出するか（氏名・スキル・希望単価等）

### ④ 技術者管理との連携
- 「登録」ボタン1つで技術者管理に自動登録するか
- 手動で内容を確認・修正してから登録するか

## 4. 技術者マーケットは必要か

### 前提：案件マーケットとの対比

```
案件マーケット（/public-projects）
  ← BPから届いた案件を掲載
  ← 技術者に「この案件どうですか」と提案する場

技術者管理（/engineers）
  ← 自社・BP技術者を社内管理するだけ
  ← 外部への公開機能なし（現状）
```

### 「誰に見せるか」で判断が変わる

**パターンA：不要（内部完結）**

技術者管理はあくまで社内台帳として使い、外部公開しないならマーケットは不要。

```
技術者メール → 技術者管理に登録
                ↓
           案件マーケットの案件と社内でマッチング
                ↓
           営業担当が提案メール送信
```

**パターンB：必要（外部公開）**

発注元企業や他のSES会社に「今こういう技術者が空いています」と公開・提案したい場合は必要。

```
技術者マーケット（/public-engineers）
  ← 「稼働可能な技術者を探している」発注元企業に見せる
  ← 「この人が空いています」と営業ツールとして使う
```

### 現状の設計について

技術者プロフィールには `is_public`（マーケット公開フラグ）がすでに存在しており、
**技術者マーケットを作る前提で設計済み。**

### ヒアリング項目

- 技術者情報を**外部（発注元）に見せたい**のか、**社内だけ**で使うのか
- 外部公開する場合、どの情報を出してどの情報を隠すか（個人名・単価等）
