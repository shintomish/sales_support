## 1. 概要

### 1.1 目的
1日5,000通規模のアウトソース（案件）メールを資産として活用し、営業の提案速度と成約率を向上させる。

### 1.2 基本思想
- `emails`テーブル（既存・保管庫）を起点に、**案件業務エンジン**を新設する
- AI抽出は段階導入（初期はルールベース、後にAI差し替え可能な設計）
- 最終判断は営業。スコアは「根拠を出す道具」であり「結論」ではない

### 1.3 全体アーキテクチャ

```
Gmail
  ↓ sync（15分毎）
emails テーブル（保管庫）
  ↓ 案件判定スコアリング
project_mail_sources テーブル（案件業務エンジン）
  ↓ マッチング
engineers テーブル（技術者）
  ↓
提案アクション（返信メール生成）
```

## 2. 案件メール判定・取込機能

### 2.1 処理フロー

```
① 除外判定（即NG）
② 強ワード判定（即OK）
③ スコア判定（グレーゾーン）
④ ステータス確定（新着 / 要確認 / 除外）
```

### 2.2 ① 除外判定（最優先・処理負荷最小）

以下のいずれかに該当する場合、即座に「除外」ステータスに確定する（本文解析なし）。

| 種別 | 条件 |
|------|------|
| メルマガ | 件名に「配信停止」「メルマガ」「広告」 |
| 請求系 | 「請求書」「お支払い」 |
| 採用系 | 「正社員募集」「中途採用」 |
| 事務連絡 | 「ご挨拶」「ご連絡」「お知らせ」 |
| 自動送信 | 差出人が `no-reply` / `noreply` |

### 2.3 ② 強ワード判定（即OK）

以下の条件を満たす場合、即座に「新着」ステータスに確定。

- **Aグループ**（業界用語）のいずれか1語 AND
- **BまたはC**のいずれか1語

| グループ | キーワード |
|---------|-----------|
| A（業界用語） | 案件、要員、技術者募集、エンジニア募集、SE募集、PG募集 |
| B（条件系） | 単価、スキル、勤務地、最寄駅、稼働、開始 |
| C（契約系） | 準委任、常駐、リモート |

### 2.4 ③ スコア判定（グレーゾーン：40〜59点が要確認）

#### スコアリング辞書（最大加点合計: 85点 → 0〜100にクランプ）

`ProjectMailScoringService.php` の実装値に準拠。キーワードリストの最新版は [540_project_mail_flow.md](./540_project_mail_flow.md) を参照。

**[A] 明示的案件ワード: +15点**

件名または本文に以下のいずれか1つが含まれる:
案件ご紹介・要員ご紹介・技術者ご紹介・案件情報・要員紹介・技術者紹介・技術者募集・エンジニア募集・SE募集・PG募集

**[B] 条件提示ワード: +10点**

件名または本文に以下のいずれか1つが含まれる:
稼働期間・稼働開始・開始時期・参画時期・稼働率・単価・月額・工数

**[C] 技術スタック: 最大+20点**

| サブカテゴリ | 加点ルール |
|---|---|
| 言語/FW | 1件目 +10点、2件目 +5点 |
| インフラ/クラウド | +5点 |
| DB | +3点 |

> 合計の上限は20点。対象キーワード:
> - **言語/FW**: Java, Spring, SpringBoot, PHP, Laravel, Python, Django, Flask, C#, .NET, JavaScript, TypeScript, React, Vue, Angular, Ruby, Rails, Go, Golang, Swift, Kotlin
> - **インフラ/クラウド**: AWS, EC2, RDS, S3, Lambda, Azure, GCP, Docker, Kubernetes, Linux
> - **DB**: MySQL, PostgreSQL, Oracle, SQLServer, MongoDB, Redis

**[D] 単価の具体性: +15点**

「○○万」という数字が本文に含まれる。

**[E] 勤務地: +10点**

以下のいずれかが含まれる: 東京・大阪・名古屋・横浜・品川・渋谷・新宿・福岡・仙台・札幌・在宅

**[F] 工程: 最大+10点**

| 工程 | 加点 |
|---|---|
| 上流(要件定義・基本設計・詳細設計) | +10点 |
| 開発系(開発・実装・製造) | +7点 |
| その他(テスト・保守・運用) | +4点 |

> 複数マッチした場合は最高点のみ採用。

**[G] 稼働・期間ワード: +5点**

即日・長期・人月 のいずれかが含まれる。

**ペナルティ(減点)**

| 区分 | 減点 | 条件 |
|---|---|---|
| 単価曖昧 | -10点 | 「スキル見合い」「応相談」が含まれる |
| 高次商流 | -10点 | 「4次」「5次」「6次」「7次」「8次」が含まれる |

**[H] ドメイン学習補正: ±20点**

同じ送信元ドメインから過去5件以上の判断実績がある場合、案件率に応じて補正:

| 案件率 | 補正 |
|---|---|
| 80%以上 | +20点 |
| 20%以下 | -20点 |
| その他 | 補正なし |

> フォームサービス(smoothcontact.com)・フリーメール(gmail.com等)は対象外。

#### 判定基準

スコアに応じて `project_mail_sources.status` の初期値が決まる。

| スコア | ステータス | 営業アクション |
|-------|-----------|--------------|
| 60点以上 | **新着** | そのまま提案フローへ進める |
| 40〜59点 | **要確認** | 営業が目視で「案件確定」か「除外」を判断 |
| 39点以下 | **除外** | 案件メールではないと判定 |

> 「要確認」→「新着」への昇格は、UI上の「案件確定」ボタンで実行される。詳細は [520_matching_flow_qa.md](./520_matching_flow_qa.md) 参照。

#### 判定結果の形式（AI差し替えを見据えた共通IF）

```json
{
  "score": 58,
  "reasons": ["Java keyword matched", "No price found"],
  "engine": "rule"
}
```

### 2.5 ④ ドメイン学習（蓄積後に有効化）

→ 2.4 の **[H] ドメイン学習補正** を参照。判断済みメール5件以上のドメインに対して案件率に応じた±20点補正を適用。

## 4. マッチングスコア設計

> ⚠️ **重要**: 現行リポジトリには **2つのマッチングサービスが並存** している。画面ごとに利用するサービスが異なるため注意。
>
> | 利用画面 | エンドポイント | サービスクラス | スコアモデル | 内訳返却 |
> |---|---|---|---|---|
> | `/matching/[id]`（案件メールから起動・主動線） | `GET /v1/project-mails/{id}/matched-engineers` | **`ProjectMailMatchingService::matchEngineers`** | **5要素100点満点**（必須40 / スキル25 / 条件20 / 稼働10 / 商流5） | あり（`breakdown` で5項目） |
> | `/public-projects/[id]` 案件詳細 | `GET /v1/matching/projects/{id}/engineers` | `MatchingService::recommendEngineers` | 4要素重み付き（skill 50% + price 25% + location 15% + availability 10%） | 4項目を個別フィールドで返却 |
> | `/engineers/[id]` 技術者詳細 | `GET /v1/matching/engineers/{id}/projects` | `MatchingService::recommendProjects` | 同上 | 同上 |
>
> 以下 4.1〜4.5 は **`MatchingService`（4要素モデル）** の仕様。`/matching/[id]` のスコア内訳はアーカイブ [425_matching_concept_archive.md](./425_matching_concept_archive.md) の5要素設計と同等で、`ProjectMailMatchingService.php` に実装されている。emails画面のプレビュー用スコア(`EmailMatchPreviewService`)とは別物。役割分担は本書「0. 本書の位置づけ」参照。

### 4.1 スコア構成（実装値）

各要素を 0〜100 点で評価し、以下の重みで合計して総合スコア（0〜100）を算出。

| 項目 | 重み | 計算根拠 |
|-----|------|------|
| ① スキル適合度       | 50% | 必須スキルの保有・経験年数 |
| ② 単価適合度         | 25% | 案件単価レンジ × 技術者希望単価レンジの重なり |
| ③ 勤務地・勤務形態   | 15% | work_style(remote/hybrid/onsite) と勤務地一致 |
| ④ 稼働可否・時期     | 10% | start_date と available_from の差分 |

### 4.2 ① スキル適合度（重み50%）

`required_skills` を以下の方式でスコアリング (`calcSkillScore`)。

- 必須スキル(`is_required = true`) は重み2、尚可スキルは重み1
- 技術者がそのスキルを持っていれば、`(experience_years / min_experience_years)` を1.0でクランプして加点
- 必須スキル未登録の案件は 100 点

### 4.3 ② 単価適合度（重み25%）

`calcPriceScore`：案件レンジ `[unit_price_min, unit_price_max]` と技術者希望レンジ `[desired_unit_price_min, desired_unit_price_max]` の重なり幅を、両レンジ平均で割ってスコア化。

- 重なりなし: 0点
- 情報不足（どちらかが null）: 50点（中間値）
- 完全に重なる: 100点

### 4.4 ③ 勤務地・勤務形態（重み15%）

`calcLocationScore`：

| 状態 | 点数 |
|---|---|
| 案件・技術者ともに `remote` | 100 |
| 一方が `remote` | 90 |
| 案件・技術者ともに `hybrid` | 80 |
| 一方が `hybrid` | 60 |
| 勤務地文字列が一部一致 (`work_location` ⇔ `preferred_location`) | 70 |
| 上記以外 | 30 |

### 4.5 ④ 稼働可否・時期（重み10%）

`calcAvailabilityScore`：技術者の `available_from` が案件 `start_date` と比べてどれだけ早いか。

| 差分(日数) | 点数 |
|-----|------|
| ≤ 0 (技術者が案件開始日までに稼働可) | 100 |
| 1〜30 日遅れ | 80 |
| 31〜60 日遅れ | 60 |
| 60日超 | `60 - (差分-60) / 3` (0でクランプ) |
| 開始日 or 稼働可能日いずれか不明 | 50 (中間値) |

### 4.6 スコアランク表示（フロント `/matching/[id]` 実装）

| スコア | 表示 | 意味 |
|-------|------|------|
| 80〜100 | ◎ | 即提案候補（緑カード） |
| 60〜79  | ○ | 有力（青カード） |
| 〜59    | △ | 参考（黄カード・デフォルト展開） |

> ※ 当初構想にあった「× 見送り」「× 非表示」は現行 FE には存在しない（全件表示）。
> ※ `/matching/[id]` の score は `ProjectMailMatchingService` の5要素合計値（最大100点）で、本ランク閾値はこの値に対して適用される。`MatchingService` の総合スコア（4要素重み付け0〜100）も同じ閾値で表示される。


### 5.2 案件→技術者マッチング画面（3秒判断）

**上部（案件サマリー・固定）**

```
Java／Spring／AWS
75〜80万｜品川（常駐）｜5月〜
外国籍不可｜年齢〜45
```

**中央（マッチング結果）**

```
◎ 最有力候補（80点以上）
  [92点] 田中｜Java5y AWS｜即日｜78万  ✔ 条件一致
  [88点] 佐藤｜Java4y    ｜2週 ｜75万  ▲ 単価▲3万

○ 候補（60〜79点）
  [76点] 鈴木｜Java3y    ｜即日｜82万  ▲ 単価＋7万

△ 参考候補（〜59点・デフォルト展開）
```

> ※ 表示数の上限・折りたたみは現行実装にはなく、全件表示・△もデフォルト展開。

**操作（カード上のボタン）**

`[提案する]` `[📧 提案メール]` `[除外]`

**技術者クリック時（スコア内訳モーダル・`/matching/[id]` 実装）**

```

稼働          10/10
商流          5/5

合計          86点
```

> ※ `/public-projects/[id]` `/engineers/[id]` の `MatchingService` 経由の画面では、`skill_match_score / price_match_score / location_match_score / availability_match_score` の4項目を直接フィールドで返す（`breakdown` は付かない）。

## 9. 実装詳細メモ（2026-04-05時点）

### 9.1 マッチング画面（/matching/[id]）実装済み仕様

- **3秒判断UI**: ランクグループ（◎即提案 / ○有力 / △検討）ごとにグリッドカード表示
- **カード表示**: 技術者ごとに稼働状況（日本語）・稼働可能日（YYYY/MM/DD形式）・スコア・スキルを表示
- **スコアモーダル**: カードクリックで内訳表示
- **参考候補（△）**: デフォルト展開（全ランク最初から表示）
- **重複ボタン削除**: 右ペインの「マッチング実行」ボタンは削除済み（「マッチング→」と重複）
- `availability_status`: `available`→**稼働可**、`scheduled`→**稼働予定**、`working`→**稼働中**

### 9.2 要確認メール一覧UI

- **Aパターン採用**: 要確認タブ選択時にフル幅の1行1判断モードを表示
- **操作**: 「案件確定」→ status=`new`へ昇格 / 「除外」→ status=`excluded`へ移動(「案件に戻す」で`new`へ復元可)
- **判定理由チップ**: スコアバッジ下に表示（技術ワード・金額・ドメイン補正）

### 9.3 ドメイン学習補正

- `ProjectMailScoringService.php` に実装済み（VPS反映済み）
- 判断済みメール5件以上のドメインに補正適用
- 案件率80%以上→+20点、20%以下→-20点
- 判定理由チップ例: `🏢 softbank.vc +20(99%/178件)`
- 「全件再スコア」ボタンで既存データに遡及適用可

### 9.4 提案メール自動生成・送信

- **草稿生成エンドポイント**:
  - `POST /api/v1/project-mails/{id}/generate-proposal`（body: `engineer_id`）案件メール画面用
  - `POST /api/v1/matching/projects/{projectId}/engineers/{engineerId}/generate-proposal` マッチング画面用
- **送信エンドポイント**:
  - `POST /api/v1/project-mails/{id}/send-proposal`（送信種別 `proposal`）
  - `POST /api/v1/matching/projects/{projectId}/engineers/{engineerId}/send-proposal`（送信種別 `matching_proposal`）
- **モデル**: Claude Haiku 4.5（`claude-haiku-4-5-20251001`）
- **書き出し固定**: `〇〇 様\n\nいつもお世話になっております。\n株式会社アイゼン・ソリューションのSES営業担当です。`
- **送信方法**: SMTP/SES 経由で送信（DeliveryCampaign + DeliverySendHistory に記録）。コピペ運用も併用可

### 9.5 「提案する」「除外」ボタンの現状

「提案する」は send-proposal API でメール送信＋履歴保存まで実行。  
「除外」はマッチング画面でのフロント状態管理のみ（永続化はマーケット連携の設計確定後）。

### 9.6 案件マーケット連携（保留）

「案件確定」→`public_projects`自動連携、「提案する」→`project_mail_proposals`保存は**保留**。
連携フローについて社内営業への説明・同意取得を経て設計確定予定。
